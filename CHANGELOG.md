## [Unreleased]

## [0.3.0] - 2026-04-18

### Added
- `nook promote <project>` command: moves an incubating project out of `lab/` into a category folder with state `active`. `--category <name>` picks the target; when omitted, `nook` prompts with the list of configured non-reserved categories. Rejects reserved names (`archived`, `shipped`, `lab`) and unknown categories. Records a `state_changed` (`incubating` → `active`) and a `category_changed` event in history, and fails with a filesystem error if a folder already exists at the destination.
- `nook ship <project>` command: moves a project's folder into `<category>/shipped/` and sets state to `shipped`. Works from `incubating`, `active`, `paused`, and `maintained`; rejects `shipped` and `archived`. `--version <version>` records the shipped version in history. Paused projects have their `paused_until` cleared.
- `nook unship <project>` command: returns a shipped project to `active`, moving its folder from `<category>/shipped/` back to `<category>/`. Rejects projects that are not shipped.
- `nook archive <project>` command: moves a project's folder into `<category>/archived/` and sets state to `archived`. Prompts for confirmation unless `--yes` is passed; declining leaves the project untouched. `--reason <text>` records a reason in history. Works from every non-archived state (including `shipped`); paused projects have their `paused_until` cleared.
- `nook unarchive <project>` command: restores an archived project to `active`, moving its folder from `<category>/archived/` back to `<category>/`. Rejects projects that are not archived.

## [0.2.0] - 2026-04-18

### Added
- `nook new <name>` command: scaffolds a new project on disk. Defaults to `lab` with state `incubating`; `--category <name>` places the project in a configured category folder with state `active` (rejects reserved names `archived`/`shipped` and unknown categories). `--scratch` marks the project for auto-prune, `--description <text>` and `--tags <list>` set initial metadata, `--template <path>` copies a local source directory into the new folder, `--fork <project>` copies an existing project (excluding its `.nook/` metadata) via the same project lookup used elsewhere. `--template` and `--fork` are mutually exclusive. On success the new project has a `.nook/project.jsonc` + `history.jsonl` with a `created` event (`source: "new"`, plus `template`/`fork` pointer when relevant) and opens in the configured editor unless `--no-open` is passed.
- `nook adopt <path>` command: brings an existing folder under nook management. The folder's basename becomes the project name (validated against the same character set as `nook new`). Defaults to category `lab` with state `incubating`; `--category <name>` picks a different (configured, non-reserved) category, and when the category isn't `lab` the default state becomes `active`. `--state <state>` overrides the default (must be one of `incubating`, `active`, `paused`, `maintained`, `shipped`, `archived`). `--description <text>` and `--tags <list>` set initial metadata. The folder is moved into `<root>/<category>/<name>` unless `--in-place` is passed; either way the folder receives a `.nook/project.jsonc` + `history.jsonl` with a `created` event (`source: "adopt"`). Refuses to adopt a folder that already contains `.nook/project.jsonc`.
- `CommandContext` now carries a `random: { next: () => number }` source so command handlers can generate project IDs deterministically under test; `main.ts` wires `Math.random` in production.

## [0.1.0] - 2026-04-18

### Added
- Initial project scaffold: `bun` runtime, strict TypeScript, `nook` CLI entry point that prints `nook` and exits.
- Pure-domain core: `Result<T, E>` type with `ok`/`err`/`isOk`/`isErr` helpers, project state union (`incubating`, `active`, `paused`, `maintained`, `shipped`, `archived`), project and config metadata type shapes, and the authoritative state-transition graph (`isValidTransition`, `allowedTargets`).
- Shared error classes: `CommandError` base plus `StateTransitionError`, `ValidationError`, and `FilesystemError` — all carrying a discriminating `code` field.
- Pure domain helpers: `generateUlid({ now, random })` (Crockford base32, 26 chars, deterministic under injected clock and rng), `isStale`/`nextStaleAtMs` (day-threshold staleness, inclusive boundary), `resolvePauseExpiry`/`isPauseExpired` (validates `--days` or `--until` and returns the resolved expiry instant), and `resolveCategoryConfig` (merges per-category overrides onto global defaults).
- Storage primitives: TypeBox schemas and `Result`-returning validators for `ProjectMetadata`, `HistoryEvent`, and `GlobalConfig`; JSONC reader with comment- and trailing-comma-tolerant parsing; atomic JSONC writer backed by `write-file-atomic`; `.gitignore`+`.nookignore` merged matcher via the `ignore` package.
- Storage modules for on-disk project state: `readProjectMetadata`/`writeProjectMetadata` for `<project>/.nook/project.jsonc` (atomic, schema-validated, creates `.nook/` on first write); `appendHistoryEvent`/`readHistoryEvents` for `<project>/.nook/history.jsonl` (append-only, preserves insertion order, returns an empty list when no history has been recorded); `readGlobalConfig`/`writeGlobalConfig` for a caller-supplied `config.jsonc` path (atomic, schema-validated, creates parent directories on first write).
- SQLite-backed project index (`storage/project-index.ts`, `bun:sqlite`): `openIndex(path)` creates the DB file and parent directories and runs idempotent migrations (`projects` table plus state/category/last_touched indexes); `upsertProject(row)` with `INSERT OR REPLACE` semantics; `queryProjects(filters)` supports `state`, `category` (single or array), `lastTouchedAfter`/`lastTouchedBefore`, returning hydrated `ProjectIndexRow`s; `markScanned(id, now)` updates `last_scanned` in place; `rebuildFromMetadata(rows)` wipes and reinserts atomically in a transaction; `isIndexRowStale(row, now, ttlMs?)` with a `DEFAULT_INDEX_TTL_MS` of five minutes.
- Filesystem modules: `walkTree(root, { ignores? })` returns forward-slash relative file paths with mtimes, skips `.git/` and `.nook/` by default, skips symlinks, and honors an injected ignores predicate; `readGitHeadTime(dir, { spawn? })` returns the HEAD commit time in milliseconds or `null` when the directory is not a git repository (spawn is injectable for testing, default uses `Bun.spawn`); `computeLastTouched({ projectDir, ignores?, cliTouchesMs?, gitSpawn? })` returns the maximum of walk mtimes, git HEAD time, and explicit CLI touches — `null` when none is present; `discoverProjects(root)` returns sorted absolute paths of every folder containing `.nook/project.jsonc`, stopping its descent at each project and skipping hidden directories and symlinks; `moveProject({ source, destination, rename?, copy? })` is atomic on same-device renames, falls back to `cp` + `rm` on `EXDEV`, creates missing destination parents, refuses to overwrite an existing destination, and rolls back a partial copy on failure.
- UI primitives: ANSI color constants, `stripAnsi`, and a `colorize(value, color, enabled)` helper in `ui/ansi-colors.ts`; `createLogger({ quiet, verbose, color, stdout, stderr })` with `info`/`warn`/`error`/`debug` that honor `--quiet` (suppresses info and warn), `--verbose` (enables debug), and `--no-color` (strips ANSI escapes); `renderTable(rows, { aligns?, separator? })` pads cells by visual width via `Bun.stringWidth`, supports per-column `"left"`/`"right"` alignment, and ignores ANSI escapes when measuring; `createSpinner({ stream, isTty, frames?, intervalMs? })` with `start`/`setText`/`stop`, animating via `\r` + braille frames in TTY mode and falling back to plain lines when `isTty` is false; thin `promptSelect`/`promptConfirm`/`promptInput` wrappers over `@inquirer/prompts` with readonly option types.
- UI rendering: `renderProjectList(sections, { color })` renders grouped project rows as a padded table with `NAME`/`STATE`/`CATEGORY`/`LAST`/`TAGS` columns, colors stale rows' state yellow and section headings bold when color is enabled, skips empty sections, and prints `No projects.` when nothing is listed; `renderStatus(summary, { color })` renders a `Projects by state:` table plus bulleted highlight blocks (e.g. stale projects, pauses expiring soon), bolds headings when color is enabled, omits zero-count states and empty highlights, and prints `Nothing to report.` for an empty summary; `isStatusEmpty(summary)` reports whether a summary has anything to show, so `nook status --quiet` can suppress output when nothing needs attention.
- Platform modules: `resolveAppPaths({ platform?, env?, homeDir? })` returns `config`, `data`, `cache`, `configFilePath` (`config.jsonc`), and `indexPath` (`state/index.sqlite`) — on Linux and macOS these follow XDG (`~/.config/nook`, `~/.local/share/nook`, `~/.cache/nook`, overridable via `XDG_*` env vars), and on Windows they resolve to `%APPDATA%\nook\Config`, `%LOCALAPPDATA%\nook\Data`, and `%LOCALAPPDATA%\nook\Cache` with homeDir-derived fallbacks; `launchEditor({ editor, projectPath, spawn? })` spawns the editor with the project path as argument and cwd, supports multi-word commands (e.g. `code --wait`), rejects empty commands, and returns `Result`; `launchAiTool({ tool, projectPath, spawn? })` spawns the AI tool with cwd set to the project path (no path argument) using inherited stdio; `openInFileManager({ projectPath, platform?, spawn? })` dispatches to `open` (macOS), `explorer` (Windows, tolerant of explorer's non-zero exit code), or `xdg-open` (Linux), and returns an error on unsupported platforms; `runAliasCommand({ command, context, platform?, spawn? })` substitutes `{path}`/`{name}`/`{id}`/`{category}` in the alias string and runs it through `sh -c` (or `cmd /c` on Windows) with cwd set to the project path, plus a standalone `substituteAliasCommand(command, context)` for callers that only need the substituted string.
- CLI plumbing: `nook --help` and `nook --version` now work end-to-end, with `--help` listing the full set of global flags (`--quiet`/`-q`, `--verbose`/`-v`, `--no-color`, `--root <path>`) and `--version` printing the package version; the binary is wired via `buildProgram({ name, version })` (commander with `--version` bound to just `--version` so `-v` stays free for `--verbose`), `createRunResult({ logger, verbose, setExitCode })` translates `Result<void, CommandError>` into stderr output plus an exit code (1 for generic failures, 2 for `validation`/`not_found`/`ambiguous_match`, cause printed via debug when `--verbose`), and `registerCommands(program, ctx)` is the flat registration point that subsequent phases extend; `CommandContext` (config, storage, ui, clock, cwd, runResult) is now assembled at startup so future command handlers can be pure and take it as an argument.
- Shell integration: `detectShell({ platform?, env? })` identifies the user's shell (bash, zsh, fish, or PowerShell) from `$SHELL` — with a Windows-specific PowerShell fallback that triggers off `$PROFILE` or `$PSModulePath` — and returns `null` for unrecognized shells; `generateSnippet(shell)` emits shell-specific `nook-cd` and `nook-ai` wrapper functions (POSIX form for bash/zsh, `function…end` for fish, `Set-Location` for PowerShell), each wrapped in `# >>> nook shell integration >>>` / `# <<< nook shell integration <<<` marker comments for round-trip detection; `findRcFile({ shell, homeDir?, env?, platform? })` resolves the rc file path (`.zshrc` honoring `$ZDOTDIR`, `.bashrc`, `config.fish` honoring `$XDG_CONFIG_HOME`, or `$PROFILE` with a sensible `Documents\PowerShell` fallback); `installRcIntegration({ rcPath, snippet })` creates or updates the rc file atomically and returns `"installed"` / `"updated"` / `"unchanged"` so repeated `nook init` runs don't duplicate the block.
- `nook config` command: `nook config path` prints the resolved config file path, `nook config cd` prints the config folder path (for shell substitution), `nook config` (no subcommand) echoes the raw `config.jsonc` bytes so user comments are preserved, `nook config get <key>` and `nook config set <key> <value>` operate on dot-notation keys (e.g. `defaults.staleness_days`, `categories.lab.staleness_days`, `aliases.zed.command`) — `set` coerces via `JSON.parse` with a string fallback, auto-creates intermediate objects under `categories.*` and `aliases.*`, rejects unknown top-level keys (so typos like `defualts.staleness_days` are caught instead of silently creating a branch), and runs TypeBox validation before writing, and `nook config edit` opens `config.jsonc` in `$EDITOR` (or `editors.default` from the config) with the editor's cwd set to the config directory.
- CLI `CommandContext` now carries `appPaths` (resolved at startup via `resolveAppPaths()`), and `ctx.ui.launchEditor` is injectable so command tests can supply a spawn-capturing fake; `launchEditor` accepts an optional `cwd` (defaults to `projectPath`, preserving existing behavior).
- `nook init` command: interactive first-run setup that walks through project root, categories (rejecting reserved names `lab`/`archived`/`shipped`), staleness threshold, scratch prune window, default editor (auto-detects `code`/`cursor`/`zed`/`nvim`/`vim` on PATH), default AI tool (auto-detects `claude`/`codex`/`opencode`/`pi`), and shell integration (detects zsh/bash/fish/PowerShell, prints the snippet, confirms append to the rc file). Nothing is written until the final summary prompt is accepted; `--force` overwrites an existing config, `--root`/`--categories` seed the prompts. On confirm, creates the project root, `<root>/lab/`, and each configured category folder, then writes `config.jsonc` atomically (with `categories.lab` seeded at 14-day staleness and `prompt_prune`) and optionally installs the rc snippet. `ctx.ui` now carries `detectBinaryOnPath`, `detectShell`, and `installRcIntegration` so command handlers can stub these side effects in tests.
- Project lookup: `nook` commands that take a `<project>` argument now resolve it against every `.nook/project.jsonc` under the configured project root, matching either the folder name (exact) or a ULID prefix (case-insensitive). When the argument matches more than one project, `nook` prompts to disambiguate; when it matches none, it returns a `not_found` error. `ctx.storage.findProject(rootDir, identifier)` is the shared storage helper backing the feature.
- `nook pause <project>` command: transitions any non-paused, non-shipped, non-archived project to `paused`, recording the resolved expiry as `paused_until` in `project.jsonc` and appending a `state_changed` history event. `--days <n>` sets the pause duration (defaults to `defaults.pause_max_days`, which is 90 after `nook init`), `--until <date>` pins an ISO date/datetime, and `--reason <text>` is recorded in history. `--days` and `--until` are mutually exclusive. The project folder is not moved.
- `nook unpause <project>` command: returns a `paused` project to `active`, clearing `paused_until` and appending a `state_changed` history event. Rejects projects that are not currently paused with a `state_transition` error.
- `nook maintain <project>` command: marks a project as `maintained` (feature complete, still receiving updates) from any of `incubating`, `active`, `paused`, or `shipped`. `--version <version>` is recorded on the history event. The project folder is not moved and any outstanding pause is cleared.
- `nook unmaintain <project>` command: returns a `maintained` project to `active` and appends a `state_changed` history event. Rejects any other state with a `state_transition` error.

### Changed
- `nook run bin/nook.ts` (and the compiled binary) now routes through commander for command parsing and global-flag handling instead of the previous placeholder that printed `nook`, awaiting each command's async action via `parseAsync` so handler output and non-zero exit codes reach the user reliably.
- `nook config --help` and `nook config get --help` / `set --help` / etc. now print per-subcommand help, auto-generated by commander; the phase-12 single-command `config [...args]` workaround has been replaced with a true nested subcommand tree. Argument parsing switched from cac to commander (six-dep allowlist unchanged in count).
